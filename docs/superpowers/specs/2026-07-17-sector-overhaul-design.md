## 1. 배경 & 목적

현재 섹터는 9개(반도체·전기전자·IT·유통·자동차·미디어·금융·방산·바이오)이고 27종에 배정돼 있으나 분포가 편중(IT 6 / 유통 6 / 나머지 2~3)돼 있고, 개인투자자에게 인기 큰 테마(에너지·2차전지·게임·소재·통신·건설·로봇 등)가 빠져 있다. 또한 섹터가 **DB CHECK 제약 · TS 유니온 타입 · 뉴스 라벨맵 3곳에 하드코딩**돼 있어 운영 중 조정이 불가능하다.

이 개편의 목적:

1. **섹터 확장·재배치** — 인기 테마를 신규 섹터로 추가하고 기존 종목 일부를 재배치, 전 섹터가 최소 2종 이상을 갖도록 균형화(27→42종, 9→18섹터).
2. **섹터를 데이터로 승격** — `sectors` 테이블 + 어드민 CRUD로 섹터 추가·이름수정·종목배치를 운영 중 수행 가능하게.
3. **섹터 뉴스 메커니즘 정립** — "좋은 섹터 뉴스라도 전부 오르진 않는다"를 반영한 **참여확률 모델**로, 체감되는 크기의 섹터 동조 움직임을 만들되 뉴스추종 지배는 막는다.
4. **섹터 뉴스 템플릿** — 섹터 × 방향 × 세기 등급별 문안 풀.
5. **자금 스케일 상향** — 초기자금 100만→**1,000만원**, 방문보너스 +10만→**+100만원/일**, 전 종목 기준가를 **실제 시장 가격대 밴드**로 재설계. (밸런스는 스케일 불변 — 순수 재denomination + 현실감)

### 비목표 (하지 않는 것)

- 가격엔진(GBM 랜덤워크·틱 σ·점프·VI·클램프·상하한±30%·수수료 0.5%) 재설계 — 손대지 않는다. 섹터 영향은 기존과 동일하게 **bias(편향) 층에서만** 주입한다. 자금·가격 스케일은 전부 배수/퍼센트 기반이라 밸런스 불변.
- tier(우량/일반/테마) 체계 변경 — 유지. 신규 종목은 사장님 지정 tier를 따른다.
- 지수 분류(NASPI/NASDAK = tier 파생) 규칙 변경 — 유지. divisor만 새 로스터·기준가로 재부트스트랩.
- 호가창·실시간 봇 — 계속 보류(PRD 결정 유지).

## 3. 섹터를 데이터로 — `sectors` 테이블 + 어드민 CRUD

### 3.1 스키마

```sql
create table sectors (
  code       text primary key,             -- 'semiconductor' 등 안정 식별자(뉴스/엔진 참조)
  label_ko   text not null,                -- '반도체' (표시·뉴스 문구)
  sort_order int  not null default 100,    -- 시세판·필터 정렬
  created_at timestamptz not null default now()
);

-- 기존 9 + 신규 9 seed
-- stocks.sector: CHECK 제약 제거 → FK로 교체
alter table stocks drop constraint stocks_sector_check;
alter table stocks
  add constraint stocks_sector_fkey
  foreign key (sector) references sectors(code);
-- NOT NULL 유지
```

- 신규 섹터 코드 9개: `energy, materials, food, cosmetics, telecom, construction, robotics, game, shipaero`.
- 삭제 가드: 종목이 참조하는 섹터는 FK가 삭제를 막음. 어드민 삭제 시 사전 검사로 친절한 에러.

### 3.2 어드민 기능 (`src/services/adminService.ts` + 콘솔 UI)

- `listSectors()` / `createSector({code,labelKo,sortOrder})`(코드 slug 검증) / `updateSector(code,{labelKo,sortOrder})`(code 불변) / `deleteSector(code)`(참조 0일 때만) / `setStockSector(stockCode, sectorCode)`(재배치).

## 4. 섹터 뉴스 메커니즘 — 참여확률 모델

### 4.1 설계 원칙

- 방향 표현은 기존과 동일하게 **bias→드리프트**(엔진 정합, 차트로 legible). 새 "확률 레이어"를 엔진에 만들지 않는다.
- "체감되는 확률"은 **화면의 % 표시가 아니라 결과(대부분 종목이 실제로 눈에 띄게 움직임)**로 드러난다.
- 섹터 구성원은 **각자 독립적으로 참여 판정** → 다수가 오르되 일부는 안 오름(현실성). 참여 판정 자체가 랜덤성을 제공하므로, 섹터 층에는 별도 방향 반전(flip)을 걸지 않는다.
- 뉴스추종 방지는 **사후/후반 노출 타이밍**으로 보장(이미 움직인 뒤 노출 → 선취 불가). 숨은 반전에 의존하지 않는다.

### 4.2 파라미터 (`src/lib/engine/bias.ts`)

| 상수 | 현행 | 신규 제안 | 의미 |
|---|---|---|---|
| 섹터 이벤트 수/일 | 50%로 1개 | 분포 추첨(평균 ≈ 1.5개) | 18섹터에서 각 섹터가 30일 내 2~3회 노출 |
| `SECTOR_MAGNITUDE` | 8 %p | **15 %p** (체감 크기, 튜닝) | 참여 종목의 상승/하락 편향 세기 |
| `SECTOR_PARTICIPATION_PROB` | 없음 | **0.7** | 구성원 각자 참여할 확률 |
| `SECTOR_UP_PROBABILITY` | 0.55 | 0.55 유지 | 섹터 이벤트의 상승 방향 확률 |

- **이벤트 수 분포(예시, 튜닝)**: {0:25%, 1:35%, 2:25%, 3:15%} → 평균 ≈ 1.3~1.5. 하루 서로 다른 섹터 다수(중복 없이).
- **섹터 선택 가중**: 기본 균등. 큰 섹터(IT 6)의 지수 과영향은 시뮬 확인 후 **크기 반비례 가중** 옵션 적용(선택).

### 4.3 알고리즘 (`bias.ts` 재작성)

```
1. drawDailyBiases → 개별 편향(individualBiases) (기존 그대로)
2. drawSectorEvents(stocks, rng): SectorEvent[]  // 위 분포로 0~3개
     각 = { sector, direction: ±1, magnitude: SECTOR_MAGNITUDE }
3. applySectorEvents(individualBiases, stocks, events):
     for 이벤트 e: for e.sector 소속 종목 s:
       if rng() < SECTOR_PARTICIPATION_PROB:
         bias[s] = clamp(bias[s] + e.direction*e.magnitude, -30, +30)
       // 미참여: 변화 없음
   → RNG 소비 순서: drawDailyBiases → drawSectorEvents → applySectorEvents,
     generateDailyPath 루프 진입 전(시드 재현성).
4. 경로: generateDailyPath(prevClose, realizeBias(bias[s], rng), tier, ...) — 기존과 동일.
5. 조기 방향뉴스 후보(pickEarlySignalTargets)는 여전히 individualBiases 기준(기존 규약 유지).
```

> **realizeBias 상호작용 — 튜닝 포인트**: 결합 편향에 realizeBias(30% flip)를 그대로 적용하면 참여 종목도 일부 하방 반전 → 체감 희석되나 밸런스 유지. `SECTOR_MAGNITUDE`(15%p)가 충분히 커서 flip 후에도 다수가 상방을 유지하도록 시뮬 검증. 체감 부족 시 (a) magnitude 상향 또는 (b) 섹터 참여분은 flip에서 제외하고 scale만 적용하는 분리 실현 도입(구현 대안).

### 4.4 섹터 뉴스 생성 (`generate.ts`)

- 하루 여러 섹터 이벤트 → **이벤트당 1건** 섹터 뉴스(stock_code=null).
- **세기 등급화**: 그 섹터 구성원의 **실현 일간 평균 등락**으로 등급(급등/강세/약세/급락). 방향은 실현 결과 기준.
- 노출: 기존과 동일 **장 후반(0.8 지점)** published_at 스탬프(사후 설명, 추종 이득 없음).
- 라벨은 `sectors.label_ko` 주입.

## 5. 섹터 뉴스 템플릿 (섹터 × 방향 × 세기)

- 구조: `SECTOR_NEWS_TEMPLATES: Record<grade, string[]>` **범용 풀**(`{sector}` 치환) + 섹터별 **플레이버 라인 소량**(선택). 18섹터 전용 풀 전면 작성은 과투자라 지양.
  - `grade ∈ { surgeUp, up, down, plungeDown }` × 각 풀.
- 캐논 준수, 이모지 금지(`no-emoji-in-ui`).
- 재사용: 개별 뉴스와 동일(제목 이력 제외 추첨, 소진 시 폴백).
- 풀 크기: 등급당 12개 내외(총 ~48) + 섹터 플레이버 점증.

## 6. 신규 종목 콘텐츠 (최대 물량)

- 신규 15종 각: 소개문 1 + **힌트 템플릿 풀 ~85개**(PRD §5.2: ±10/0 각 15, ±20/±30 각 10) → 약 **1,275개**.
- `src/lib/news/templates.ts` `HINT_TEMPLATES`에 15개 코드 키 추가. `DISCLOSURE_TEMPLATES`는 범용(`{name}`/`{pct}`)이라 신규 불필요.
- 구현 계획에서 종목 단위 분할(에이전트 병렬 생성 후 캐논 검수).

## 7. 지수 · 밸런스 · 시뮬레이션

- **지수 divisor 재부트스트랩**: 42종 기준일(2026-07-31) **신규 기준가** 시총으로 NASPI/NASDAK = 1,000pt 되도록 재계산(roster_27.sql:86-94 패턴).
- **이벤트 밀도**: `EVENT_RATIO=0.15` 종목 수 비례 → 42종에서 개별 이벤트 ≈ 6개/일 자동 스케일.
- **`npm run simulate` 재검증 (필수)**:
  - 초기자금 1,000만 반영(결과 배수는 불변 예상).
  - 섹터 상관 증가 → 꼬리 확대 예상. "섹터 바스켓 추종"이 지배 전략 아닌지 확인.
  - 튜닝 노브: `SECTOR_MAGNITUDE`, `SECTOR_PARTICIPATION_PROB`, 이벤트 수 분포, (필요 시) 크기 반비례 가중.
  - 시뮬레이터가 섹터를 새 데이터 소스에서 로드하도록 반영.
- 목표(PRD §10): 1위 3~10배 구간·역전 가능성 유지, 뉴스추종 비지배.

## 10. 리스크 · 오픈 이슈

- **체감 vs 밸런스 트레이드오프**(§4.3): magnitude/참여율/flip 상호작용은 시뮬로만 확정 — 1차 값은 제안치, 반드시 재측정.
- **콘텐츠 물량**(§6): ~1,275 힌트 템플릿이 최대 비용. 일정 압박 시 신규 종목 풀 크기 축소(폴백 규칙으로 가능).
- **tier 가격 하한 폐지**(§2.1): tier는 이제 변동성 클래스만 의미. 밴드 겹침 허용 — 표시/필터에서 tier·가격을 혼동하지 않게 UI 문구 점검.
- **StockSector 유니온 완화**(§3.3): `string`화로 컴파일타임 안전성 일부 상실 — 섹터 오타는 런타임/FK로만 검출.
- **종목 코드 확정**: 신규 15종 4자 코드는 구현 시 캐논·중복검사로 확정.
