**작성일:** 2026-07-18
**브랜치:** `worktree-feat+onboarding` (워크트리)
**상위 스펙:** `docs/superpowers/specs/2026-07-18-immersion-features-design.md` §④
**선행 완료:** 출석 스트릭(PR#42), 소셜확장 3종(PR#43). **병렬 진행:** 스티커(`feat/stickers`), 배지·타이틀(`feat/badges`)
**로드맵상 위치:** 마지막(다른 기능이 있어야 툴팁 붙일 대상이 생김 — 상위 스펙 "구현 순서" 4)

## 목적

처음 접속한 손님이 규칙을 몰라 이탈하지 않게, **딱 필요한 지점에서만** 짧게 안내한다.
풀 인터랙티브 튜토리얼(단계별 코치마크·강제 진행)은 이 이벤트 규모에 과함(YAGNI). 대신 두 가지만 한다:

1. **상황별 툴팁** — 상하한·VI·서킷브레이커·매도 수수료·지정가 등 오해하기 쉬운 지표 옆에 "?" 팝오버로 한 줄 설명. `/guide`의 정적 규칙을 **필요한 지점에 분산**해서, 손님이 규칙 페이지로 이탈하지 않고 그 자리에서 이해하게 한다.
2. **첫 매수 축하** — 유저의 첫 매수 체결 순간에 축하 연출을 띄워 "나 방금 뭔가 했다"는 유능감을 즉시 만든다. 튜토리얼의 결승선 역할.

- 온보딩은 **현금가치 0** → 밸런스·시뮬 영향 없음. 공정성 안전.
- UI 문구는 [[no-emoji-in-ui]] 준수(유니코드 이모지 금지), 트리거 아이콘은 lucide 개별 임포트.

## 핵심 결정 (상위 방향 확정 + 본 스펙 구체화)

1. **풀 튜토리얼 배제.** 오버레이 코치마크·강제 스텝·진행률 바 없음. 상황별 툴팁 + 첫 매수 축하 **2종만**.
2. **툴팁 텍스트는 `/guide`와 단일 출처 공유.** 규칙 문구를 두 곳(가이드 페이지·툴팁)에 복붙하지 않는다. 현재 `src/app/guide/page.tsx`에 인라인된 `RULES` 배열을 공유 모듈로 추출해 가이드 페이지와 `InfoTip`이 같은 문자열을 임포트한다(내용 드리프트 방지).
3. **재사용 가능한 `InfoTip` 하나로 통일.** 지점마다 다른 컴포넌트를 만들지 않고, `label`+본문을 받는 단일 `InfoTip`을 여러 곳에 배치.
4. **첫 매수 판정은 서버 권위 신호(권장).** 클라이언트가 "처음인지"를 추측하지 않고 `execute_trade`가 `isFirstBuy`를 반환 → 축하 트리거. 한번 참이면 다시 참일 수 없어 **중복 방지 자동**. (대안: `users.onboarded_at`, 또는 프론트 휴리스틱 — §데이터 모델 트레이드오프 참고. **최종안은 사장님 확인** — §열린 결정)
5. **첫 매수 축하 연출은 기존 자산 우선 재사용.** 이미 있는 `TradeSuccessOverlay`(체크 드로우 + 미니 폭죽)를 **첫 매수 전용 변형**으로 확장하는 방향을 우선 검토(신규 모달 최소화). 최종 연출 형태는 열린 결정.

## 상황별 툴팁

기존 코드에서 실제로 렌더되는 지표에 매핑했다:

| # | 지점 | 파일 | 툴팁 내용(요지) | 대응 `/guide` 규칙 |
|---|------|------|----------------|-------------------|
| 1 | **상한가·하한가** 행 | `src/components/quotes/StockStats.tsx` (`rows`의 상한가/하한가) | 하루 등락폭은 전일 종가 ±30%로 제한 | "사고팔기" |
| 2 | **서킷브레이커** 배너 | `src/components/quotes/MarketHaltBanner.tsx` | 시장 전체 요동 시 운영자가 전 종목 매매를 잠시 정지 | "거래 정지" |
| 3 | **VI(종목 거래정지)** 표시 | 종목 상세 정지 표시 지점 (`src/app/stocks/[code]/page.tsx`·`TradePanel` 정지 상태) | 주가 급변 시 그 종목만 5분 자동 정지 | "거래 정지" |
| 4 | **매도 수수료 0.5%** | `src/components/trade/TradePanel.tsx` (매도 다이얼로그 예상 체결액 근처) | 매도 시에만 0.5% 수수료, 매수는 무료 | "사고팔기" |
| 5 | **지정가 예약주문** | `TradePanel.tsx` 지정가 탭 | 지정한 가격 도달 시 자동 체결, 장 마감에 소멸, 최대 10건 | "지정가 예약주문" |

- 1·4·5는 우선순위 상(거래 직결·오해 잦음), 2·3은 발동 시에만 노출되므로 후순위. **개장 마감(7/28) 압박 시 1·4·5 먼저, 2·3은 후속**으로 잘라도 된다.
- 배지/방문보너스/장 시간 등 나머지 규칙은 이미 화면·`/guide`에 충분히 노출되어 툴팁 불필요(YAGNI).

### `/guide` 정적 규칙 재사용 방식

현재 `src/app/guide/page.tsx`의 `RULES` 배열은 **페이지 내부에 인라인**되어 있어 툴팁에서 임포트할 수 없다. 최소 리팩터링:

- 규칙 본문 문자열을 공유 모듈(예: `src/lib/guide/rules.ts` 또는 `src/content/guide.ts`)로 추출하고 **키를 부여**(`FEE`, `PRICE_LIMIT`, `VI`, `CIRCUIT_BREAKER`, `LIMIT_ORDER` 등).
- `guide/page.tsx`는 그 모듈을 매핑해 기존과 동일하게 렌더(문안·아이콘 그대로, 시각적 변화 0).
- `InfoTip`은 같은 키로 짧은 요약 문구를 참조. 가이드 본문이 길면 툴팁용 **한 줄 요약**을 별 필드로 병기(가이드=상세, 툴팁=요약, 출처 파일 하나).
- 이 추출은 **문안 변경 없음** → 회귀 위험 낮음. 커밋 단위 분리 권장(리팩터 → 기능).

### 컴포넌트 설계

- **`InfoTip`** (신규, `src/components/common/InfoTip.tsx`, `"use client"`)
  - 트리거: lucide `CircleHelpIcon`(개별 임포트) 아이콘 버튼, 접근성 `aria-label`.
  - 콘텐츠: shadcn `Popover`에 짧은 텍스트. 모바일 탭·데스크톱 클릭 모두 팝오버가 `Tooltip`보다 안정적(터치 hover 부재) → **`Popover` 채택**.
  - props: `{ label: string; children: ReactNode }` 또는 `{ ruleKey }`로 공유 모듈 조회.
  - 인라인 배치용으로 작게(예: `size-3.5` 아이콘, muted 색).
- **shadcn `Popover` 프리미티브 필요.** 현재 `src/components/ui/`에 `popover.tsx` **없음**(`dialog.tsx`만 존재) → `src/components/ui/popover.tsx` + radix `@radix-ui/react-popover` 의존성 추가 필요. **주의: 스티커 스펙의 `StickerPicker`도 동일 `Popover`를 요구** → 병렬 구현 시 파일·package.json 충돌(§표면 겹침).

## 첫 매수 축하

### 판정 방식 (트레이드오프)

| 방식 | 변경 범위 | 권위 | 중복 방지 | 배지 연계 | 비고 |
|------|----------|------|-----------|-----------|------|
| **A. 프론트 휴리스틱** (거래내역 존재 or `localStorage`) | 마이그레이션 0, 프론트만 | 낮음(추측) | 약함(기기별·스토리지 삭제 시 재발) | 어려움 | 순수 연출이라 오작동해도 무해하지만 다기기서 반복 노출 |
| **B. `execute_trade`가 `isFirstBuy` 반환** (권장) | **함수 replace 마이그레이션 1종**(테이블/컬럼 변경 없음) | 높음(서버) | **자동**(한번 참이면 영구 거짓) | 쉬움(같은 신호 공유) | 리허설 데이터 **재생성 불필요**(함수만 교체) |
| **C. `users.onboarded_at` timestamptz** | **스키마 마이그레이션 1종**(컬럼 추가) | 높음 | 자동(non-null 게이트) | 쉬움·명시적 | 랭킹·프로필 등에서 조회/조인 가능. 다만 온보딩 외 용도가 없으면 과설계 |

- **권장: B.** `execute_trade`가 insert **직전** `not exists (select 1 from trades where user_id = p_user_id and side = 'buy')`를 계산해 반환 jsonb에 `isFirstBuy` 추가. 스키마·데이터 변경 없어 리허설 재생성 불필요, 배지 로직과 신호 공유 용이.
- "첫 **매수**"(side='buy') 기준 vs "첫 **거래**"(매도 포함) 기준은 열린 결정. 초기 자금만 있는 신규 유저는 매수가 첫 행동이므로 매수 기준이 자연스럽다.

### 트리거 지점

- 프론트 단일 퍼널: `src/components/trade/TradePanel.tsx`의 `onSuccess` 콜백(시장가 `useSubmitTrade`, 지정가 즉시체결 `useSubmitOrder` 모두 이 콜백으로 수렴 → `setSuccess`).
- `TradeResult` DTO(`src/services/tradeService.ts`)에 `isFirstBuy: boolean` 추가 → `onSuccess`가 이 값을 보고 **일반 체결 팝업 대신 첫 매수 축하 변형**을 띄운다.
- 지정가 **예약**(즉시 체결 아님)은 첫 매수로 치지 않는다(아직 미체결). 예약이 나중에 배치/장중 로직으로 체결될 때의 첫 매수 판정은 B안이면 그 체결 경로도 `execute_trade`를 타므로 자연 커버(단, 그 경로엔 축하 UI 트리거가 없음 → 연출은 "유저가 직접 낸 즉시 체결"에서만, 판정 플래그는 데이터로 정확 — 열린 결정에서 정리).

### 연출 (형태는 열린 결정)

- **우선안:** 기존 `TradeSuccessOverlay`(체크 드로우 + 미니 폭죽, 2초 자동 닫힘)를 재사용하되 `isFirstBuy`일 때 **특별 변형**(더 큰 축하 문구 + 세계관 톤 한 줄, [[naraka-lore-canon]] 어휘 준수). 신규 컴포넌트 최소화.
- **대안:** 별도 축하 모달(dialog) 또는 sonner 토스트. 모달은 주목도 높지만 흐름 끊김, 토스트는 가벼움.
- **중복 방지:** B/C 방식이면 서버 신호가 1회성 보장(2번째 매수엔 `isFirstBuy=false`) → 별도 클라이언트 상태 불필요.

## 데이터 모델

- **권장(B): 순수 스키마 마이그레이션 없음.** `execute_trade` 함수만 `create or replace`로 교체해 반환 jsonb에 `isFirstBuy` 추가(테이블·컬럼·인덱스 변경 0). → prod `db push`는 필요하나 **리허설 데이터 재생성 불필요**([[rehearsal-reset-before-open]] 무관).
- **대안(C): 마이그레이션 1종** — `alter table users add column onboarded_at timestamptz null;` + `execute_trade`가 첫 매수 시 `update ... set onboarded_at = now() where onboarded_at is null` 세팅. 컬럼 추가라 cascade·reset 규약 그대로.
- **대안(A): 마이그레이션 0** — 프론트만. 서버 신뢰성 포기(연출 한정이라 허용 가능하나 다기기 반복).

## 서비스 / API / UI 계층

### 서비스
- `src/services/tradeService.ts` — `TradeResult`에 `isFirstBuy: boolean` 추가(B/C안). RPC 반환값 그대로 매핑. 그 외 로직 무변경.
- `execute_trade`(DB 함수) — B안 시 insert 직전 첫 매수 여부 계산 후 반환 jsonb에 포함. C안 시 `onboarded_at` 세팅 병행. **단일 트랜잭션 원칙 유지**(아키텍처 원칙 1).

### API
- 신규 엔드포인트 **없음**. 매수 체결 API 응답 DTO에 `isFirstBuy`가 자연 포함(기존 `ApiResponse<T>` 래퍼 그대로).
- 툴팁은 정적 문자열이라 API 불필요.

### UI
- `InfoTip`(신규, `src/components/common/`) + `Popover` 프리미티브(신규, `src/components/ui/popover.tsx`).
- 공유 규칙 모듈(신규, `src/lib/guide/rules.ts` 등) + `guide/page.tsx` 리팩터(문안 무변경).
- 툴팁 5개 지점 배치(위 표) — 각 대상 컴포넌트에 `<InfoTip>` 삽입(레이아웃 최소 변경).
- `TradeSuccessOverlay` 확장(첫 매수 변형) + `TradePanel.onSuccess`에서 분기.

## 범위 밖 (YAGNI / 후속)

- 단계별 코치마크·강제 스텝·진행률 바·스킵 버튼 — 하지 않음.
- 웹 푸시/PWA 온보딩, 이메일 환영 — 이벤트 특성상 없음(이메일 미수집).
- "따라 담기"·미니 리그 등 유도 기능 — 상위 스펙 제외 그대로.
- 툴팁 A/B·노출 로깅·분석 — 이벤트 규모에 과함.
- 온보딩 완료 배지 자체 정의 — 배지 스펙(§③) 소관. 여기선 `isFirstBuy` 신호만 제공.

## 다른 기능과의 표면 겹침 (병렬 구현 시 충돌 지점)

| 표면 | 겹치는 기능 | 위험 | 조정안 |

## 열린 결정 (사장님 확인 필요 — 임의 확정 금지)

1. **툴팁 붙일 최종 지점.** 후보 5종(상하한·서킷브레이커·VI·매도 수수료·지정가) 중 개장 전 어디까지 넣을지. 최소안(수수료·상하한·지정가) vs 전체.
2. **첫 매수 판정 방식.** A(프론트 휴리스틱·마이그레이션 0) / **B(`execute_trade` `isFirstBuy` 반환·함수 replace·권장)** / C(`users.onboarded_at`·스키마 변경). 배지 재사용·리허설 부담·명시성 트레이드오프.
3. **첫 매수 축하 연출 형태.** 기존 `TradeSuccessOverlay` 첫 매수 변형(권장) / 별도 모달 / sonner 토스트.
4. **첫 "매수" vs 첫 "거래" 기준.** 매도 포함 여부(신규 유저는 매수가 첫 행동이라 매수 기준이 자연스러움).
5. **배지와의 연출 통합.** 첫 매수 축하가 "첫 거래 배지" 부여와 겹칠 때 이중 노출을 어떻게 합칠지(축하 안에 배지 표기 vs 순차 노출 vs 배지 미부여). 배지 스펙 확정과 함께 결정.
