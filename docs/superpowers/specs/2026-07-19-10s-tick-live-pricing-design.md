# 10초 틱 실시간 시세 + 견적-잠금 거래 설계

- 작성일: 2026-07-19
- 브랜치: `feat/10s-tick-live-pricing`
- 상태: 설계 확정(구현 계획 대기)

## 1. 문제 정의

현재 상세 페이지 헤드라인 가격은 `usePriceWiggle`로 **2.5초마다 ±0.1% 무작위 진동**하는 연출값(`displayPrice`)을 보여준다. 그러나 체결은 서버의 고정 5분 틱값(`quote.price`)으로 이뤄진다. 결과적으로 **"보이는 가격에 사려는데 다른 고정 가격에 체결"** 되는 불일치가 발생한다.

근본 원인: wiggle이 `Math.random()` 기반이라 **서버가 재현할 수 없는 노이즈**이고, 그래서 체결은 틱값을 쓸 수밖에 없다. 화면과 체결이 구조적으로 갈라진다.

## 2. 목표

- **생동감**: 가격이 실제로 자주 움직이고, 틱 사이도 wiggle로 부드럽게 살아있다.
- **체결 정직성**: 사용자가 거래를 확정하는 값과 실제 체결값이 정확히 일치한다.
- **예측 불가능 + 조작 방지**: 미래 가격을 미리 알 수 없어 인트라틱 차익이 불가능하다.
- **진짜 5분 캔들차트**: OHLC가 채워진 캔들스틱.

### 비목표 (유지되는 불변식)

- 모든 돈 계산은 서버 단일 트랜잭션(아키텍처 원칙 1). 클라이언트가 보낸 가격은 절대 신뢰하지 않는다.
- 가격은 사전 생성 경로(원칙 2). 장중에는 읽기만.
- 자산은 정수(원)(원칙 3).

## 3. 핵심 개념 — 결정론 ≠ 예측 가능

서버가 필요한 것은 **재현 가능성**(체결 시각의 값을 다시 계산/조회 가능)이고, 사용자가 그 값을 **미리 알 수 있느냐(예측 가능성)** 는 별개다.

물리 법칙: 클라이언트는 사용자의 기계다. 클라이언트가 화면값을 스스로 "계산"하면 그 공식을 읽은 사용자는 미래값도 계산할 수 있다(예측 가능 → 차익). 따라서 예측 불가능성을 지키려면 **클라이언트가 미래를 계산하게 두지 않고, 서버가 시간이 될 때만 값을 흘려준다.**

→ 진동을 "클라이언트 랜덤 공식"이 아니라 **"배치가 숨은 시드로 미리 생성해 두고, 현재 시각까지만 공개하는 진짜 10초 틱"** 으로 바꾼다. 미래 틱은 응답에서 제외한다.

## 4. 아키텍처

### 4.1 데이터 모델

**`daily_ticks` — 10초 간격으로 전환 (원자 진실)**

- 장중 12h 기준 종목당 **4,320틱/일** (`tick_index` 0 ~ 4,319, smallint 범위 내).
- 기존 CHECK 제약(`between 0 and 83`/`143`) → 상한을 장 시간·틱 간격에서 파생하도록 확대.
- 체결·현재가가 읽는 **단일 진실 소스**.
- 컬럼은 기존 유지: `(stock_code, date, tick_index, price, is_halted, volume)`.

**`daily_candles` — 5분 OHLC (신규, 차트 전용)**

```sql
create table daily_candles (
  stock_code text not null references stocks (code),
  date date not null,
  bucket smallint not null,        -- 5분 버킷 인덱스 (0 ~ 143)
  open bigint not null,
  high bigint not null,
  low bigint not null,
  close bigint not null,
  volume bigint not null default 0,
  primary key (stock_code, date, bucket)
);
create index daily_candles_date_bucket_idx on daily_candles (date, bucket);
```

- 배치가 10초틱 30개(300s/10s)를 굴려 사전 집계.
- 차트는 이 테이블만 읽는다 → 8,640/4,320행 조회로 인한 PostgREST 1,000행 페이지네이션 회피, 기존 차트 코드 최소 변경.

### 4.2 틱 인덱스 산출

```
tick_index = floor(장중경과초 / TICK_INTERVAL_SECONDS)   -- TICK_INTERVAL_SECONDS = 10
bucket     = floor(tick_index / (300 / TICK_INTERVAL_SECONDS))  -- 5분 버킷 = 30틱
```

`TICK_INTERVAL_SECONDS`는 **config 값**으로 노출해 10초 ↔ (폴백) 이전 값 전환이 가능하게 한다.

### 4.3 가격 공개·체결 흐름

1. **현재가 API**: `floor(장중경과초/10)` 인덱스의 틱 행을 읽어 반환. **미래 인덱스는 절대 반환하지 않는다.** 응답에 `asOf`(서버 시각) 포함.
2. **체결 RPC**: 매수/매도 실행 시 서버가 **주문 시점의 현재 틱 행**을 읽어 그 price로 체결. 클라이언트가 보낸 가격은 무시.
3. → 화면(견적) = 서버 틱값 = 체결값. 자동 일치.

시드는 배치 서버 전용이고 미래 틱은 미공개이므로, 사용자가 할 수 있는 최선은 "방금 공개된 값 보고 지금 주문"뿐 → 인트라틱 예측·차익 소멸.

### 4.4 연출 3층 (역할 분리)

| 층 | 정체 | 역할 | 정직성 |
|----|------|------|--------|
| 1. 진짜 움직임 | 10초 실제 틱 갱신 | 10초마다 값 변경 + **플래시** | 실값 |
| 2. wiggle(유지) | 현재 10초 틱값 **주위 ±0.1%** 잔진동 | 10초 사이 정적을 메워 실크한 생동감 | 표시용(±0.1%만) |
| 3. 거래 | 견적-잠금 다이얼로그 | 실제 체결 | 실값 |

- wiggle은 `usePriceWiggle`을 거의 그대로 사용. 입력 `base`에 10초 틱값을 넣으면 새 틱마다 자동 재앵커(현재도 `base` 변경 시 무효화 구조).
- **플래시는 진짜 10초 틱 변화에만** 발동(wiggle이 사이를 메우므로 저가 종목 정적 문제 없음).
- wiggle은 순수 표시용이며, **거래는 반드시 견적-잠금 값으로만** 체결된다.

### 4.5 견적-잠금 UI (슬리피지 해결)

10초 모델에선 진짜로 값이 자주 변하므로, 사용자가 늦게 누르면 실제로 다른(진짜) 값에 체결될 수 있다(정직한 슬리피지). 이를 UX로 흡수한다.

- 매수/매도 다이얼로그를 열면 **서버 현재 틱값(실값, wiggle 아님)** 을 크게 고정 표시: `현재가 12,340원 · 10초 후 갱신`.
- 10초 경계에서 값이 바뀌면 견적 갱신:
  - **유리한 변동**(매수: 하락 / 매도: 상승) → 자동 반영.
  - **불리한 변동**(매수: 상승 / 매도: 하락) → **재확인 요구**(버튼 상태 리셋, "가격이 바뀌었어요 · 다시 확인").
- 체결은 항상 서버가 주문 시점 틱값으로 실행(서버 권위 불변).

## 5. 밸런스 재유도

10초 = 5분의 30배 틱. 일변동성을 보존하려면 틱당 파라미터를 스케일한다.

- `σ_10s ≈ σ_5min / √30`
- `drift_10s ≈ drift_5min / 30`
- 랜덤워크는 **부동소수 원값으로 진행**하고, 저장·체결 시에만 반올림(다음 스텝은 원값에서 계산) → 4,320회 반올림 드리프트 누적 방지.
- `npm run simulate`로 일변동성·중앙값 드리프트가 기존과 동등한지 검증(별도 Task).

## 6. 하류 로직 일반화

"마지막 틱 = tick_index 143" 가정을 제거하고 **`MAX(tick_index)`** 또는 캔들 close로 일반화한다.

- **밴드(±30%)**: 직전 개장일 종가 = 마지막 틱 close.
- **지수(나스피/나스닥)**: 종목가 가중합 — 틱 해상도 10초로 증가, 지수 캔들(5분)도 집계.
- **거래량**: 틱당 10초 버킷. 캔들 volume = 30틱 합.
- **VI(서킷브레이커) 정지**: 정지 구간은 **벽시계 분 단위로 정의**하고 행 범위(예: 2분 = 12틱)로 확장. `is_halted` 플래그는 틱 단위 유지.

## 7. 운영·비용 보완

- **폴링**: 현재가 전용 경량 엔드포인트에 **엣지 캐시 `s-maxage=10`** → 모든 유저 CDN 히트, 오리진 계산 10초당 1회 공유. 탭 숨김 시 폴링 정지(`refetchIntervalInBackground: false` + visibility), 복귀 시 재개.
- **배치**: 하루 181k행(42종목) 생성·삽입을 RPC 내부 **SQL `generate_series`** 로 처리(jsonb 페이로드 폭탄·pg_net 타임아웃 회피).
- **저장**: 캔들 집계 완료 후 오래된 **raw 10초틱 프루닝**(캔들·최근 N일만 유지, 캔들은 영구 보관). 월 ~5.4M행 → 관리 가능.
- **차트 장기 뷰**: 다중 해상도 — 인트라데이 5분봉(`daily_candles`), 월간 뷰 일봉. 범위별 쿼리 분리.
- **클라 애니메이션**: CSS 플래시 유지, 트레일링 이징은 상세 등 보이는 화면만.

## 8. 롤아웃 안전장치

- `TICK_INTERVAL_SECONDS`를 **config 값**으로 빼 10초 ↔ 폴백 즉시 전환.
- `daily_candles`는 순수 추가라 기존 경로를 깨지 않는다.
- 리허설 환경에서 검증(틱 부트스트랩·리허설 초기화 활용). 개장 8/1 전 리허설 필수.
- 워크트리 분리 원칙에 따라 `feat/10s-tick-live-pricing` 브랜치에서 진행.
