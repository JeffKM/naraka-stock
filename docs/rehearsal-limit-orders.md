# 리허설 체크리스트 — 지정가 예약주문 (Phase 10 · PRD §4.5)

리허설 장(07-15~, 프로덕션·실제 5분 틱·실시간)에서 확인할 지정가 예약주문 점검표.
개발 중 SQL 시나리오·브라우저 E2E(11/11)로 검증됐으나, **실시간 소급 체결(C)과 폐장 배치 정산(F)** 은 실환경에서만 온전히 확인되므로 리허설에서 최종 점검한다.

핵심만 보면: **C(실시간 소급 체결)와 F(폐장 정산)** 둘이 눈으로 확인되면 지정가는 완결.

## A. 기본 접수 UI
- [ ] 종목 상세 → 구매/판매 → **지정가 탭** 전환 시 지정가 입력·현재가/±나눔칩·"가능 범위 N~N원"(±30% 밴드) 노출
- [ ] 밴드 밖 지정가 입력 → "상하한가 범위를 벗어났어요" + 예약 버튼 **비활성**
- [ ] 매수=금액 입력 / 매도=수량+비율퀵(전량·50%·25%)

## B. 즉시 시장가 체결 (조건 이미 충족)
- [ ] **매수 지정가 ≥ 현재가**로 접수 → "즉시 시장가로 체결돼요" 안내 → 제출 시 **현재 틱값**으로 즉시 체결(성공 오버레이), 예약 목록엔 안 남음
- [ ] 매도 지정가 ≤ 현재가도 동일하게 즉시 체결

## C. 대기 예약 → 실시간 소급 체결 ★핵심
- [ ] 매수 지정가를 **현재가보다 조금 아래**(다음 몇 틱 안에 닿을 만한 값)로 접수 → "예약됐어요" 토스트, MyOrdersCard에 대기로 표시
- [ ] 5~15분(1~3틱) 지나 가격이 지정가를 통과한 뒤 **지갑/종목 상세를 열면**(폴링/재접속) → 자동 체결되어 보유·현금 반영, 예약 내역에 "체결"
- [ ] 체결가가 **지정가로 정확히** 기록됐는지(틱값이 지정가보다 유리해도 지정가 고정)
- [ ] 체결 시각(내역)이 실제 조건 닿은 **과거 틱 시각**으로 소급됐는지
- [ ] 앱을 닫아둔 채 조건 통과 후 다시 열어도 동일하게 체결(접속 빈도와 무관 = 공정)

> 랜덤 시세라 "안 닿을" 수도 있음. 확정 확인이 필요하면 **현재가 바로 아래(직전 틱 근처)** 로 걸거나, 운영자 SQL(H)로 강제 검증.

## D. 예약 락 = 주문가능 차감
- [ ] 매수 대기 예약을 걸면 지갑/매수창 **"주문 가능" 현금이 예약금만큼 감소**
- [ ] 그 상태에서 예약금 포함 초과 금액으로 **시장가 매수 시도 → 잔고 부족**(예약분은 못 씀)
- [ ] 매도 대기 예약 시 그 종목 **매도 가능 수량이 예약분만큼 감소**("보유 N주(예약 M주 제외)")
- [ ] **총자산 표시는 예약과 무관하게 그대로**(예약은 실제 이동 아님 → 랭킹 불변)

## E. 취소·만료
- [ ] 대기 예약 **취소** → 즉시 사라지고 주문가능 현금/수량 원복(내역엔 안 남김)
- [ ] **미체결 10건 상한** — 11번째 접수 시 거부 안내

## F. 폐장 배치 정산 (자정 24:00 이후)
- [ ] 폐장까지 조건 미충족인 대기 예약이 **자동 만료**되고 주문가능이 원복됐는지(다음날 아침 지갑 확인)
- [ ] 폐장 순간 조건이 충족돼 있던 예약은 **체결**로 마감됐는지
- [ ] 배당(금요일)·순위가 정산 후 보유 기준으로 맞는지(정산이 배당보다 먼저 실행됨)

## G. 정지 상황 (선택 — 발동 시)
- [ ] VI 거래정지 틱 구간에선 그 틱에서 체결되지 않고 **다음 정상 틱**에서 체결
- [ ] 서킷브레이커(운영자 수동) 중엔 지정가 접수·정산 보류, 해제 후 정상 처리

## H. 운영자 확인 (Supabase SQL Editor — 지상검증)
```sql
-- 특정 유저 예약/체결 현황
select id, side, limit_price, status,
       coalesce(reserved_cash::text, reserved_qty::text) reserved,
       filled_price, filled_at at time zone 'Asia/Seoul' filled_kst
from orders
where user_id = (select id from users where nickname = '닉네임')
order by id desc;

-- 전체 대기 예약 수 / 잠긴 현금 총액 (고아 예약 점검)
select count(*) pending, coalesce(sum(reserved_cash), 0) locked_cash
from orders where status = 'pending';
```
- [ ] 폐장 후 `status='pending'`이 **남아있지 않은지**(모두 filled/expired) — 고아 예약이 현금을 계속 묶지 않는지

> 시간 안 기다리고 강제 검증이 필요하면, SQL Editor에서 `place_limit_order(...)`·`settle_limit_orders(user_id, p_at => '2026-07-15 20:00:00+09')`를 `p_at` 오버라이드로 호출해 특정 시각 기준 소급 체결을 즉석에서 재현할 수 있음.

## 리허설 데이터 초기화 시 주의
- 콘솔 "리허설 데이터 초기화"(`reset_rehearsal_data`)는 **orders도 함께 삭제**한다(장 운영 설정은 유지).
- 수동 SQL로 초기화한다면 `delete from orders;`를 반드시 포함(고아 예약이 현금을 묶는 것 방지).
