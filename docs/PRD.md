- 이벤트 기간: **2026-08-01(토) ~ 2026-08-30(일)** — 정기 휴장 없음(매일 개장), 개장일 30일
  - 마지막 개장일은 이벤트 종료일 **08-30(일)**. 순위도 이날 장 마감 종가로 확정

## 1. 목표

### 1.1 비즈니스 목표
- 카페 방문 유도: 가입 코드·방문 보너스를 매장에서만 발급 → 참여하려면 매장에 와야 함

- 커뮤니티 재미: 손님들 사이에 "오늘 뭐 샀냐" 대화거리 제공

### 1.2 제품 목표
- 캐주얼 유저도 3분 안에 이해하는 단순한 거래 시스템 (시장가 매매만)

## 4. 가격 엔진 — "하루 먼저 뽑는" 사전 생성 경로 방식

시나리오를 미리 정하지 않되 뉴스 힌트를 가능하게 하는 핵심 설계.

### 4.1 일일 사이클

```
폐장 시각 직후 (일일 배치, 1회 — pg_cron이 설정된 폐장 시각으로 트리거):
1. 오늘 종가 확정, 정산 (금요일이면 배당 지급)
2. 내일 각 종목의 "일일 편향(bias)" 랜덤 추첨
   - 매일 2~4개 종목에 이벤트 배정: {+30, +20, +10, -10, -20, -30}% 중 가중 추첨
   - 나머지 종목은 중립 (등급별 기본 변동성만)
3. 내일 개장~폐장의 가격 경로를 전부 사전 생성해 DB 저장
   - 틱 수는 장 시간에서 파생 (5분 간격 — 예: 12:00~24:00이면 144틱)
   - 편향을 드리프트로 갖는 랜덤워크, 직전 개장일 종가 ±30% 클램프
   - 경로 생성 시 VI(거래정지) 구간도 함께 계산해 저장
4. 내일 경로의 "실제 움직임"으로부터 정식뉴스 자동 생성
   - 각 뉴스의 published_at을 해당 움직임이 나오는 틱 시각으로 스탬프
     → 내일 장중에 시간차로 하나씩 노출 (§5.1)
5. 오늘 실제 결과로부터 "공시" 뉴스 자동 생성 (published_at = 폐장 시각)
```

**휴장일 처리 (기본: 정기 휴장 없음 — 매일 개장)**
- 정기 휴장 요일은 없다(`config.closed_weekdays = []`). 휴장일이 필요해지면 추후 논의해 `config`로 지정한다. 아래 조건부 로직은 코드에 그대로 남아 있어, 휴장일을 지정하는 순간 자동으로 동작한다.
- 배치는 매일 폐장 시각에 실행하되, 휴장일이 지정된 경우 조건부로 동작:
  - 정산(종가 확정·배당)은 **당일이 개장일일 때만** 수행
  - 경로·뉴스 생성은 **익일이 개장일일 때만** 수행 — 익일이 휴장이면 건너뛰고, 기준가(±30% 클램프 기준)는 직전 개장일 종가를 쓴다
- 폐장 시각은 어드민 설정값(`config.market_close_hour`)을 따르며, 변경 시 배치 크론 스케줄도 자동 재조정된다(`reschedule_daily_batch`). 폐장이 24:00이면 배치는 다음 날 00:00(KST)에 돌고, "방금 닫힌 날"을 어제로 계산한다
- 개장일 판정: `config`의 휴장 요일(`closed_weekdays`)·임시 휴무(`holiday_exceptions`)·특별 개장일(`extra_open_days`) 규칙으로 결정 (기본은 전부 개장)
- 휴장일 UI(휴장일 지정 시): 장 상태 "휴장" 표시, 매매 차단(서버 검증 포함), 차트·랭킹 열람은 가능

### 4.2 장중 동작
- 현재가 = 사전 생성 경로에서 `floor(현재시각 / 5분)` 위치의 틱 값 → **장중 크론 불필요**, 서버는 읽기만 함
- 매수/매도는 항상 "현재 틱 가격"으로 서버에서 체결 (클라이언트가 보내는 가격은 신뢰하지 않음)
- **금액 기준 소수점 체결**(토스 벤치마킹): 매수는 금액(원)만 받아 `수량 = 금액 / 체결가`(6자리 절사)로 체결하고 대금만 차감(남는 원은 현금 유지) → 자본 100% 투입 가능. 매도는 수량·금액 택일이며, 금액모드 초과분·반올림 잔량은 전량으로 스냅. 클라이언트는 금액 또는 수량만 전송하고 수량 환산·대금 계산은 전부 `execute_trade`가 수행
- 프론트엔드는 틱 사이 5분 동안 가격을 미세하게 흔들리는 것처럼 **보간 연출** (실제 체결가는 틱 값 고정)
- **정식뉴스도 사전 생성 경로처럼 시간차로 노출**: 배치가 published_at을 미래 틱 시각으로 스탬프해 두고, 피드는 `published_at <= now()`만 조회 → 장중 크론 없이 뉴스가 저절로 풀린다

### 4.3 변동성 완화 장치 (연출 요소)
- **점프 이벤트**: 경로 생성 시 낮은 확률(우량 0.2%/일반 0.5%/테마주 1.5% per 틱)로 한 틱에 ±2~7% 급변 주입 — 차트 재미 + VI 발동 재료
- **VI (종목별)**: 직전 틱 대비 **±6% 이상 급변** → 해당 종목 5분(1틱) 거래정지 + 화면 배너 (초안 "10분 내 ±10%"는 발동 확률 0이라 시뮬레이션 후 조정, 2026-07-12)
- **서킷브레이커 (시장 전체)**: 어드민이 수동 발동 → 전 종목 10분 거래정지, 사이트 전체 경고 연출. 발동 시 남은 당일 경로를 현재 지점부터 재생성 가능 (깜짝 폭락/폭등 이벤트용)

### 4.4 상한가/하한가
- 모든 틱 가격은 `직전 개장일 종가 × [0.7, 1.3]` 범위로 클램프
- 상한가/하한가 도달 시 해당 방향 진행 불가 (매매 자체는 가능)
- UI: 상한가 도달 종목은 빨간 배지 "上", 하한가는 파란 배지 "下" 연출

## 5. 뉴스 시스템

### 5.1 뉴스 3등급 (발행 모델 개편 2026-07-14)

| 등급 | 적중률 | 내용 | 발행 주체·시점 |
|------|--------|------|----------------|
| 📰 공시 | 100% (과거 사실) | 그날 실제 급등락(±5%↑ 또는 상·하한) 결과 요약 | 자동 · **폐장 시각** |
| 📢 정식 뉴스 | 90% | 그날 경로의 실제 움직임을 설명 (10% 확률로 방향 오보) | 자동 · **장중, 움직임 나오는 틱 시각** |
| 💬 찌라시 | 55% | 근거 없는 소문 (어드민이 흘리고 그에 맞춰 시세 조정) | **어드민 수동** · 즉시 |

- **정식뉴스는 예측 힌트가 아니라 "설명형"**: 사전 생성 경로에서 가장 가파른 움직임이 나오는 틱 시각에 published_at을 스탬프해, 그 움직임과 **동시에** 노출된다(§4.2). 10%는 실제와 반대 방향 오보. 예측 선취가 불가능하므로 "뉴스만 따라가면 승리"가 원천 차단된다(§10)
- 유의미한 움직임(|일간 등락|≥5%)이라도 반드시 뉴스가 나오지는 않음 (커버리지 ~70%) → 뉴스 없는 급등도 존재. 잔잔한 종목엔 가끔 방향성 없는 잡뉴스만
- **찌라시는 자동 생성하지 않는다** — 어드민이 콘솔에서 직접 발행하고 시세를 손으로 조정하는 라이브 연출 도구 (§9 수동 뉴스)
- 공시·정식뉴스는 `is_auto=true`라 배치 재실행 시 교체됨, 어드민 수동 뉴스(찌라시 포함)는 보존

### 5.2 뉴스 본문 생성
- 종목 × 사건 세기(+30/+20/+10/중립/-10/-20/-30)별 **한국어 템플릿 풀**을 미리 작성 (나라카 세계관 플레이버)
- 정식뉴스는 경로의 실제 일간 등락 크기로 세기(±10/20/30)를, 실제 방향(90%)으로 부호를 정해 템플릿을 고른다
- 풀 크기 (사장님 확정 2026-07-14): **±10/0 등급 각 15개, ±20/±30 등급 각 10개** (종목당 85개, 27종 총 2,295개)
- **한번 발행에 쓴 템플릿은 재사용 금지** — 배치가 발행 이력(제목 기준)을 제외하고 추첨. 풀 소진 시에만 전체 풀 폴백 (이벤트 기간 내 소진은 사실상 불가능)
- 일일 배치가 템플릿에서 랜덤 선택 + 변수 치환 → 운영 손 안 감
- 어드민이 수동 뉴스 작성도 가능 (특별 이벤트 공지 등)

### 9.1 아키텍처 핵심 결정
- **장중 크론 없음**: 5분 틱은 사전 생성 경로를 읽기만 하므로 Vercel Hobby(크론 1일 1회 제한)로도 충분
- **배치 구조** (2026-07-12 확정): 경로 생성·편향 추첨 로직은 **TS 엔진**(`src/lib/engine/`, 시드 RNG로 결정적) — 몬테카를로 시뮬레이션과 운영 배치가 같은 코드를 쓴다. DB 반영은 `apply_daily_batch()` 단일 트랜잭션(멱등). 트리거는 pg_cron→pg_net HTTP 호출로 `/api/cron/daily-batch` (CRON_SECRET 인증, 등록은 T-702 배포 시)
- **모든 돈 계산은 서버(Postgres 함수/트랜잭션)에서**: 매수/매도는 단일 트랜잭션으로 잔고 검증 → 체결 → 기록. 클라이언트는 표시만
- 돈 컬럼(현금·가격·대금·수수료·평단)은 정수(원 단위) — 부동소수점 금지. **보유·거래 수량만 numeric(20,6) 소수점**(소수점 주식) — numeric은 정확한 십진수라 부동소수점 아님
- **인증은 커스텀(닉네임+비밀번호), Supabase Auth 미사용**: 모든 DB 접근은 서버(service role) 경유, RLS는 전 테이블 기본 차단 — 클라이언트 직접 접근 봉쇄
