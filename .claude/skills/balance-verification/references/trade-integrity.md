# 거래 무결성 검증 참조

trade-integrity-verifier가 로드한다. 거래 경로가 서버 단일 트랜잭션에서 조작 불가능하게 돈을 계산하는지 검증한다.

## 검증 대상 (Postgres 함수)

- `execute_trade(...)` — 시장가 매수/매도. `p_at` 파라미터로 장중 시각 오버라이드 가능(API 미노출).
- `place_limit_order(...)`·`settle_limit_orders(...)` — 지정가 등록·체결. 마이그: `20260714070000_limit_orders`.
- 관련: `20260714040000_fractional_shares`(소수주), `20260714060000_sell_fee_50bp`(매도 수수료 0.5%), `20260714100000_admin_cash_adjust`.

## 로컬 검증 환경

```bash
npx supabase start          # Docker 필요
npx supabase db reset       # 마이그 + seed.sql 적용
# 이후 psql로 함수 직접 호출 (p_at으로 장중 시각 오버라이드)
```

## 체크리스트

1. **단일 트랜잭션 원자성**: 잔고 검증 → 현재 틱 가격 체결 → 기록이 하나의 트랜잭션인가. 중간 실패 시 전체 롤백되는가(부분 커밋 없음).
2. **클라이언트 가격 불신뢰**: 체결가가 클라이언트 입력이 아니라 **서버의 현재 틱 값**으로 강제되는가. 함수 인자로 가격을 받아 그대로 쓰는 경로가 없는가.
3. **잔고 검증**: 잔고 부족 직전/경계에서 음수 잔고나 초과 매수가 발생하지 않는가.
4. **매도 수수료 0.5%**: 정확히 1회 부과되는가(이중/누락 없음). 반올림 방향이 일관적인가.
5. **소수주 반올림**: fractional 체결에서 반올림 누적으로 원(整数)이 생성·소멸되지 않는가 — 총합 보존 확인.
6. **정수(원) 불변식**: 모든 금액이 정수인가. 부동소수점이 끼어드는 경로가 없는가.
7. **경합**: 동시 매수/매도(동일 유저·동일 틱)에서 잔고·보유수량 정합성이 깨지지 않는가.
8. **상하한 체결**: 상한가/하한가 틱에서 체결가가 밴드를 벗어나지 않는가(엔진의 밴드 계산과 일치).

## 재현 SQL 패턴

```sql
-- p_at으로 특정 장중 시각의 체결가 검증
select execute_trade(
  p_user := '<uuid>', p_stock := '<id>', p_side := 'buy',
  p_quantity := 10, p_at := '2026-08-05 15:30:00+09'
);
-- 총합 보존: 거래 전후 (현금 + 보유평가액 + 수수료누계) 불변 확인
```

## 조작 시나리오 (반드시 시도)

- 클라이언트가 유리한 가격을 함수에 주입 시도 → 서버가 무시하는지.
- 잔고보다 큰 매수 → 거부되는지.
- 반올림을 이용한 미세 차익 반복 → 총합에서 누수가 잡히는지.
